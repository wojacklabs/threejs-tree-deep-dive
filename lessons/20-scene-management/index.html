왜 구조화가 필요한가?

지금까지의 레슨은 모든 코드를 하나의 파일에 작성했습니다. 실제 프로젝트에서는 씬이 커지면 관리가 불가능해집니다.

  소규모 데모:          대규모 프로젝트:
  main.ts (300줄)      src/
                       ├── main.ts (진입점)
                       ├── World.ts (씬 관리)
                       ├── rooms/
                       │   ├── Room.ts (방 클래스)
                       │   ├── RedRoom.ts
                       │   └── BlueRoom.ts
                       ├── loaders/
                       │   └── AssetManager.ts
                       └── utils/
                           └── dispose.ts

클래스로 씬 분리

이 레슨에서는 Room 클래스로 각 방을 독립적으로 관리합니다:

class Room {
  group: THREE.Group
  private meshes: THREE.Mesh[] = []

  constructor(name, position, size, color) {
    this.group = new THREE.Group()
    this.build()
  }

  build() { /* mesh 생성 */ }
  update(t) { /* 애니메이션 */ }
  dispose() { /* 리소스 해제 */ }
}

각 방이 자신의 mesh, light, 애니메이션을 관리합니다. 방을 추가/제거할 때 다른 코드에 영향이 없습니다.

LoadingManager

여러 리소스(모델, 텍스처 등)의 로딩을 한 곳에서 관리합니다:

const manager = new THREE.LoadingManager()

manager.onStart = (url, loaded, total) => {
  showLoadingScreen()
}
manager.onProgress = (url, loaded, total) => {
  updateProgressBar(loaded / total)
}
manager.onLoad = () => {
  hideLoadingScreen()
}
manager.onError = (url) => {
  showError(url)
}

// 모든 로더에 manager 연결
const texLoader = new THREE.TextureLoader(manager)
const gltfLoader = new GLTFLoader(manager)
// → 모든 로딩이 끝나면 onLoad 호출

캔버스의 로딩 화면이 이 패턴입니다. 모든 리소스가 로드되면 자동으로 사라집니다.

LoadingManager를 사용하지 않으면 각 리소스마다 개별적으로 로딩 상태를 추적해야 합니다. manager가 전체 진행률을 자동 계산합니다.

dispose() — 메모리 해제

Three.js에서 가장 흔한 버그는 메모리 누수입니다. 씬에서 remove()만 하면 GPU 메모리에 남아있습니다:

// ✗ 메모리 누수!
scene.remove(mesh)  // 씬에서만 제거, GPU에 남아있음

// ✓ 올바른 해제
mesh.geometry.dispose()    // geometry 해제
mesh.material.dispose()    // material 해제
// material에 텍스처가 있으면:
mesh.material.map?.dispose()
mesh.material.normalMap?.dispose()
scene.remove(mesh)         // 씬에서 제거

lil-gui에서 Dispose Red Room을 클릭하고 info bar의 geometries 수가 줄어드는 것을 확인하세요.

renderer.info.memory로 현재 GPU에 있는 geometry/texture 수를 확인할 수 있습니다. dispose 전후를 비교하면 누수를 감지할 수 있습니다.

dispose 체크리스트

  리소스              dispose 방법
  ───────────────────────────────────
  BufferGeometry     geometry.dispose()
  Material           material.dispose()
  Texture            texture.dispose()
  RenderTarget       renderTarget.dispose()
  Shadow Map         light.shadow.map.dispose()
  EffectComposer     각 pass의 dispose()
  OrbitControls      controls.dispose()
  EventListener      removeEventListener()

visible vs dispose

// 잠시 숨기기 (나중에 다시 보일 것)
mesh.visible = false
// GPU 메모리 유지, 즉시 복원 가능

// 영구 제거 (다시 안 씀)
mesh.geometry.dispose()
mesh.material.dispose()
scene.remove(mesh)
// GPU 메모리 해제, 복원 불가

lil-gui의 Rooms 폴더는 visible 토글 (메모리 유지), Memory 폴더의 Dispose는 완전 해제입니다.

renderer.info

렌더링 상태를 모니터링하는 핵심 도구입니다:

renderer.info.render.calls      // draw call 수
renderer.info.render.triangles  // 삼각형 수
renderer.info.memory.geometries // GPU geometry 수
renderer.info.memory.textures   // GPU 텍스처 수

info bar에서 이 값들을 실시간으로 확인할 수 있습니다. 방을 dispose하면 geometries가 줄어드는 것을 관찰하세요.

거리 기반 단계별 전략

visible과 dispose는 양자택일이 아니라 거리에 따라 단계적으로 적용합니다:

  거리        전략               메모리     복원 비용
  ───────────────────────────────────────────────
  ~근처       visible = true     유지       즉시
  ~중거리     visible = false    유지       즉시
  ~원거리     LOD 저폴리 교체    일부 해제   빠름
  ~매우 멀리  dispose            완전 해제   재로드 (수 초)

function updateByDistance(house, playerPos) {
  const dist = house.position.distanceTo(playerPos)

  if (dist < 20)       // 근처: 풀 디테일
    house.interior.visible = true
  else if (dist < 50)  // 중거리: 내부 숨김
    house.interior.visible = false
  else if (dist < 100) // 원거리: 저폴리만
    house.setLOD('low')
  else                 // 매우 멀리: 해제
    house.dispose()
}

다시 돌아올 가능성이 높으면 visible, 확실히 안 돌아오면 dispose가 원칙입니다. visible = false는 렌더링을 안 하지만 GPU 메모리는 여전히 차지하므로, 모든 것을 숨기기만 하면 메모리가 부족해집니다.

오픈월드 + 건물 내부 분리

실제 게임에서 매우 흔한 패턴입니다. 두 가지 접근법:

  방식 1: 동적 로드 (GTA 방식)
  문에 다가가면 → 백그라운드 로드 → 끊김 없이 진입
  장점: 심리스   단점: 구현 복잡

  방식 2: 씬 교체 (스카이림 방식)
  문을 열면 → 로딩 화면 → 다른 씬으로 전환
  장점: 간단, 메모리 분리   단점: 로딩 화면

// 방식 2: 씬 교체
const worldScene = new THREE.Scene()  // 오픈월드
const houseScene = new THREE.Scene()  // 집 내부
let activeScene = worldScene

function enterHouse() { activeScene = houseScene }
function exitHouse()  { activeScene = worldScene }

// 렌더 루프 — 활성 씬만 렌더링
renderer.render(activeScene, camera)

핵심은 "보이지 않는 것의 리소스를 해제하여 보이는 것에 집중"하는 것입니다. 씬 분리는 기기 성능을 늘리는 게 아니라, 한정된 성능을 필요한 곳에만 집중시킵니다.

  • renderer.info — 메모리/draw call 모니터링

2단계: Room 클래스

class Room {
  group: THREE.Group
  private meshes: THREE.Mesh[] = []
  private lights: THREE.Light[] = []
  private disposed = false

  constructor(name, position, size, color) {
    this.group = new THREE.Group()
    this.group.position.copy(position)
    this.build()
  }

  private createMesh(geo, mat): THREE.Mesh {
    const mesh = new THREE.Mesh(geo, mat)
    this.group.add(mesh)
    this.meshes.push(mesh)  // 추적
    return mesh
  }

  build() {
    // 바닥, 벽, 오브젝트, 조명 생성
    // createMesh로 생성 → meshes 배열에 자동 추적
  }

  update(t: number) {
    // 매 프레임 애니메이션
  }

  dispose() {
    if (this.disposed) return
    this.disposed = true
    for (const mesh of this.meshes) {
      mesh.geometry.dispose()
      mesh.material.dispose()
    }
    for (const light of this.lights) {
      light.shadow?.map?.dispose()
    }
    this.meshes.length = 0
  }
}

disposed 플래그로 중복 dispose를 방지합니다. meshes 배열로 생성한 리소스를 추적하여 빠짐없이 해제합니다.

4단계: dispose로 방 제거

function disposeRoom(index: number) {
  const room = rooms[index]
  if (!room) return

  room.dispose()            // geometry/material 해제
  scene.remove(room.group)  // 씬에서 제거
  rooms[index] = null       // 참조 해제

  // renderer.info.memory.geometries가 줄어든 것 확인
}
