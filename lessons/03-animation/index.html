렌더 루프란?

Three.js에서 애니메이션은 매 프레임마다 상태를 바꾸고 다시 그리는 것입니다. 이를 렌더 루프(render loop)라고 합니다.

  렌더 루프의 흐름:

  animate() 호출
      │
      ├── 1. 다음 프레임 예약 (requestAnimationFrame)
      ├── 2. 오브젝트 상태 변경 (위치, 회전, 스케일 등)
      └── 3. 렌더링 (renderer.render)
      │
      ▼
  다음 프레임에 animate() 다시 호출
  (보통 초당 60회 = 60fps)

requestAnimationFrame

requestAnimationFrame(callback)은 브라우저가 다음 화면을 그리기 직전에 콜백을 실행해달라고 예약하는 API입니다.

setInterval과의 차이:

  • setInterval — 탭이 비활성이어도 계속 실행. GPU/CPU 낭비
  • requestAnimationFrame — 탭이 비활성이면 자동 정지. 브라우저 리페인트와 동기화
Three.js 애니메이션에는 항상 requestAnimationFrame을 사용합니다. setInterval이나 setTimeout은 사용하지 않습니다.

프레임 기반 vs 시간 기반

애니메이션 속도를 결정하는 두 가지 방식이 있습니다:

방식 1: 프레임 기반 (비추천)

cube.rotation.y += 0.01  // 매 프레임 0.01 라디안

문제: 60fps 모니터에서는 초당 0.6rad, 144fps 모니터에서는 초당 1.44rad. 프레임 속도에 따라 애니메이션 속도가 달라집니다.

방식 2: 시간 기반 (추천)

cube.rotation.y = elapsed * 0.6  // 경과 시간에 비례

60fps든 144fps든 30fps든, 1초 후에는 항상 0.6rad. 어떤 기기에서든 동일한 속도.

  60fps 모니터:   |--|--|--|--|--|--|  (프레임 60개/초)
  144fps 모니터:  |-|-|-|-|-|-|-|-|  (프레임 144개/초)

  프레임 기반: 144fps가 2.4배 빠르게 움직임 ✗
  시간 기반:   둘 다 같은 속도로 움직임     ✓

THREE.Clock

Three.js는 시간 관리를 위한 Clock 클래스를 제공합니다:

  • clock.getElapsedTime() — Clock 생성 이후 총 경과 시간 (초)
  • clock.getDelta() — 이전 호출 이후 경과 시간 (초). 보통 0.016초 (60fps 기준)
주의: getElapsedTime()과 getDelta()를 같은 프레임에서 둘 다 호출하면 내부 타이머가 꼬입니다. 둘 중 하나만 사용하거나, getElapsedTime()만 쓰고 delta는 직접 계산하는 것이 안전합니다.

삼각함수로 애니메이션

Math.sin()과 Math.cos()는 3D 애니메이션의 핵심 도구입니다:

sin() — 바운스, 떠다니기

// -1 ~ +1 사이를 부드럽게 왕복
sphere.position.y = 1 + Math.sin(elapsed * 2) * 0.8
//                  ↑기준점   ↑속도 (빠르게)    ↑진폭 (범위)

cos() + sin() — 원형 궤도

sphere.position.x = centerX + Math.cos(elapsed) * radius
sphere.position.z = centerZ + Math.sin(elapsed) * radius
  sin/cos를 조합하면 원형 궤도:

         cos
    ───────────→ x
    │   ╭─────╮
    │  ╱       ╲
sin │ │    ●    │  ← 중심점
    │  ╲       ╱
    ↓   ╰─────╯
    z

  x = center + cos(t) * r
  z = center + sin(t) * r

scale 펄스

const s = 1 + Math.sin(elapsed * 3) * 0.3
mesh.scale.set(s, s, s)  // 0.7 ~ 1.3 사이로 펄스

파동 합성 — 리사주 궤적

각 축에 다른 주파수의 sin/cos을 적용하면 복잡한 궤적이 만들어집니다. 이를 리사주(Lissajous) 곡선이라 합니다:

// x, y, z에 각각 다른 주파수
mesh.position.x = Math.sin(t * 1.0) * 1.2
mesh.position.y = Math.sin(t * 1.7) * 0.8
mesh.position.z = Math.cos(t * 0.8) * 1.0

주파수 비율이 정수비(1:2, 2:3)면 궤적이 반복되고, 무리수비(1:1.7)면 반복되지 않아 더 자연스럽습니다. 캔버스의 Lissajous 데모에서 초록색 궤적선을 확인해보세요.

파동 합성 — 복합 파동

같은 축에 여러 주파수의 sin을 합산하면 불규칙하지만 유기적인 움직임이 됩니다. 이것이 푸리에 합성의 원리입니다:

// 기본 파동 + 중간 디테일 + 미세 진동
mesh.position.y = Math.sin(t * 1.0) * 0.4    // 기본
               + Math.sin(t * 2.3) * 0.2     // 중간 디테일
               + Math.sin(t * 4.7) * 0.05    // 미세 진동

주파수가 높아질수록 진폭을 줄이는 것이 핵심입니다. 캔버스의 wave composite 데모가 이 방식입니다.

목적별 최적 애니메이션 기법

어떤 움직임을 구현할지에 따라 적합한 방식이 다릅니다:

  목적                      최적 기법
  ─────────────────────────────────────────
  일정 속도 회전             elapsed * speed
  위아래 떠다니기            sin(t) (단일 주파수)
  원형/타원 궤도             cos(t), sin(t)
  자연스러운 idle 모션       sin 여러 개 합성 (무리수비)
  복잡한 3D 궤적            리사주 (축별 다른 주파수)
  좌우 왕복 (핑퐁)           delta * direction
  파도/지형 변형             sin 합성 or Perlin noise
  카메라 흔들림 (핸드헬드)    sin 합성 (작은 진폭, 높은 주파수)
  호흡/심장박동              sin + sin (비대칭 조합)
  예측 불가 자연 움직임       Perlin/Simplex noise
sin 합성과 Perlin noise는 원리가 같습니다. Perlin noise 내부에서 하는 일이 본질적으로 여러 주파수(octaves)의 합성이며, 일일이 주파수를 설계하지 않아도 되는 편의성 때문에 노이즈를 더 선호합니다. Perlin noise는 Lesson 20(절차적 생성)에서 다룹니다.

delta 기반 이동

위치를 매 프레임 조금씩 이동시킬 때는 delta를 곱해야 합니다:

// 초당 2 유닛 이동
cube.position.x += direction * speed * delta
//                              2      0.016 (60fps)

delta를 곱하면 프레임 속도와 무관하게 일정한 이동 속도를 보장합니다. 핑퐁(왕복) 이동은 방향을 반전시키면 됩니다:

if (cube.position.x > 1) direction = -1
if (cube.position.x < -2) direction = 1

캔버스 데모 요약

왼쪽 캔버스에서 8가지 애니메이션 패턴을 확인할 수 있습니다:

  • rotation += 0.01 — 프레임 기반 회전 (비추천)
  • Clock elapsed — 시간 기반 회전 (추천)
  • sin() bounce — 삼각함수 바운스
  • circular orbit — cos/sin 원형 궤도
  • scale pulse — 스케일 펄스
  • Lissajous — 축별 다른 주파수, 초록 궤적선
  • wave composite — 여러 sin 합성, 자연스러운 떠다니기
  • delta ping-pong — delta 기반 왕복 이동

전체 코드

1단계: Clock 생성

const clock = new THREE.Clock()
// 생성 시점부터 시간 측정 시작

Clock은 내부적으로 performance.now()를 사용하여 밀리초 단위의 정밀한 시간을 측정합니다.

2단계: 렌더 루프에서 시간 가져오기

function animate() {
  requestAnimationFrame(animate)

  const elapsed = clock.getElapsedTime()  // 총 경과 시간 (초)
  const delta = elapsed - prevTime        // 직접 계산 (getDelta 혼용 방지)
  prevTime = elapsed

  // ... 여기서 오브젝트 상태 변경 ...

  controls.update()
  renderer.render(scene, camera)
}
animate()

3단계: 프레임 기반 회전 (비추천)

// 매 프레임 고정값을 더함
cube1.rotation.x += 0.01
cube1.rotation.y += 0.01

단순하지만, 프레임 속도가 다른 기기에서 애니메이션 속도가 달라집니다. 프로토타이핑에만 사용하세요.

4단계: 시간 기반 회전 (추천)

// 경과 시간에 비례 — 어떤 fps에서든 동일한 속도
cube2.rotation.x = elapsed * 0.6
cube2.rotation.y = elapsed * 0.6

elapsed는 계속 증가하는 값이므로, +=가 아닌 =으로 직접 대입합니다. 이 방식은 "t초일 때 이 각도"가 결정적(deterministic)이라는 장점도 있습니다.

5단계: sin() 바운스

// sin은 -1 ~ +1 을 왕복
// 기준점 1, 진폭 0.8 → 0.2 ~ 1.8 범위로 왕복
bounceSphere.position.y = 1 + Math.sin(elapsed * 2) * 0.8
//                        ↑     ↑            ↑        ↑
//                      기준   함수        속도(×2)   진폭

elapsed * 2에서 2를 키우면 더 빠르게, 줄이면 더 느리게 왕복합니다.

6단계: 원형 궤도

// cos → x축, sin → z축 → 원형 경로
orbitSphere.position.x = 3.5 + Math.cos(elapsed * 1.5) * 1.5
orbitSphere.position.z = Math.sin(elapsed * 1.5) * 1.5
orbitSphere.position.y = 0.3

// 궤도 경로 시각화 (Line)
const orbitPath = new THREE.Line(
  new THREE.BufferGeometry().setFromPoints(
    Array.from({ length: 65 }, (_, i) => {
      const angle = (i / 64) * Math.PI * 2
      return new THREE.Vector3(
        3.5 + Math.cos(angle) * 1.5,
        0.01,
        Math.sin(angle) * 1.5
      )
    })
  ),
  new THREE.LineBasicMaterial({ color: 0x58a6ff })
)

궤도 중심은 (3.5, 0, 0), 반지름은 1.5입니다. Line으로 경로를 시각화하면 궤도를 눈으로 확인할 수 있습니다.

7단계: 스케일 펄스

const scale = 1 + Math.sin(elapsed * 3) * 0.3
pulseTorus.scale.set(scale, scale, scale)
// x, y, z 모두 같은 값 → 균일 스케일링 (형태 유지)
// 0.7 ~ 1.3 사이로 펄스

8단계: delta 기반 핑퐁

let deltaDirection = 1

// animate() 내부:
deltaCube.position.x += deltaDirection * 2 * delta
//                      방향(±1)       속도   프레임 보정

if (deltaCube.position.x > 1) deltaDirection = -1
if (deltaCube.position.x < -2) deltaDirection = 1

delta (약 0.016초)를 곱하는 이유: 60fps에서는 2 × 0.016 = 0.032씩 이동, 30fps에서는 2 × 0.033 = 0.066씩 이동. 프레임이 적으면 더 많이 이동해서 결과적으로 같은 속도가 됩니다.
